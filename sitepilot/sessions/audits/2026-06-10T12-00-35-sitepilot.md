## Decisions made

The synthesizing agent selected Design 2 (Fact Harness, read-only projection using `ai_fact_point`) as the consensus winner across all three judge verdicts (3–0). The final plan grafts specific mechanisms from Designs 1 and 3 onto the Design 2 skeleton.

Key architectural decisions in the synthesized plan:

- The projection (`ai_fact_point`) is strictly read-only and derived; only `lib/facts/` may write it, enforced by a CI grep gate.
- Manual and API overrides are stored in a separate capture-tier pin table (`ai_fact_pin`) rather than inside the projection, preserving truncate-and-rebuild disposability.
- Write-through refresh (`refreshFacts`) must call the same `resolveCell()` function as `rebuild.js`; divergence is impossible by construction and asserted by unit test.
- Write consistency is framed honestly as same-request best-effort plus nightly reconcile, not same-transaction (the server-side `q()` helper is per-statement with no transaction wrapper).
- The source-kind set is frozen at six (`venture_column`, `field_value`, `task_scalar`, `task_anchor`, `stage_anchor`, `site_visit`); no seventh kind without a design document.
- Bounded coerce (`date_parse`, `numeric_parse` only) is included to handle TEXT-captured dates; no transform DSL.
- `unit` column is added to `am_fact_def` to avoid key-encoding units (e.g., `system_size_kw`), which is the documented Salesforce anti-pattern.
- API facts payload is map-keyed (`facts:{installation_start_date:{…}}`) rather than array-shaped.
- `UNBOUND` state is synthesized at read time and is distinct from `EMPTY` (bound but no value) on the wire.
- `ai_fact_point_log` (append-only history) ships in Phase 2 alongside the single writer.
- Migration 164 (minted-once `internal_name` on `am_stage_blueprint` and `am_assignment_blueprint`) is included as a non-blocking Phase 6 follow-on.
- Money-column disambiguation uses a designation policy: only `contract_value → ai_venture.contract_value` is registered initially; `monetary_value`, `total_price`, and `revenue` are deliberately unregistered.
- Design 3's canonical-as-system-of-record architecture is explicitly rejected due to non-recomputability when out-of-band script-tier capture writers exist.
- Read-through rendering of bound fields is explicitly rejected (would violate the mig-158 freeze and break persistence-by-value specs 80/81).
- Provisioner pre-binds shipped template blueprints at tenant creation so a new tenant's digest works on day one.
- Blueprint-version-clone binding copy is registered now as a written obligation in `docs/BLUEPRINT_VERSIONING_AND_INHERITANCE.md` rather than claimed as an existing chokepoint (mig 160 contains no clone code; FV is unshipped).
- PATCH on `/api/v1/projects/:id/facts` writes `ai_fact_pin`, never the projection or capture; response carries `state:"PINNED"`; adopts Design 3's `409 stale_write` and `422` error semantics; uses the existing `am_api_idempotency` machinery from mig 159.
