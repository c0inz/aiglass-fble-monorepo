## Decisions made

The operator (owner/senior dev) commissioned a complete, opinionated architecture design for a project-level fact layer for SitePilot. The resulting design is named the **Fact Harness** and was produced in full during this session. The following design decisions are embedded in the delivered output:

- The architecture bias adopted is projection/read-model-first (CQRS-lite / semantic-layer flavored): capture stays exactly as-is; a governed projection (`ai_fact_point`) and a fact registry (`am_fact_def`, `am_fact_binding`) are layered on top.
- Three new tables are specified: `am_fact_def` (registry of immutable fact keys), `am_fact_binding` (ID-based source declarations with survivorship priority), and `ai_fact_point` (materialized projection, one row per venture/fact).
- Fact keys are immutable snake_case names, never derived from labels at runtime; a starter catalog of ~35 solar-EPC facts is defined in code (`lib/facts/catalog.js`).
- Label inference is permitted exactly once, at declaration time, via a suggestion wizard on the Settings → Project Facts UI page, with a human confirming; never at runtime.
- The `am_field_spec` dedup-merge (DATA_SHAPE §4.7 hard prerequisite for key indexing) is explicitly **not** a prerequisite for the Fact Harness; duplicate specs are handled by binding both IDs at different priorities.
- Write-through refresh is synchronous and transaction-scoped at four existing chokepoints: form save, property save, task-virtual dual-write funnel, and venture-column PATCH handler; plus stage/step lifecycle transitions for anchor/REF facts.
- The `rebuild.js` (truncate-and-rebuild per tenant per fact) script must exist before anything else ships; it is the recovery path for all drift bugs.
- Agent API facts endpoints are scoped into v1 (no EAV, permanent columns rule) because the fact projection is a typed, permanent-schema surface. Endpoints specified: `GET /api/v1/meta/facts`, `GET /api/v1/projects/:id/facts`, `GET /api/v1/projects/:id/facts/:key`, and `GET /api/v1/projects` with `fact.*` filter params.
- `PUT /api/v1/projects/:id/facts/:key` does not write the projection; it routes to the capture surface (venture-column PATCH, or writeValues when the shape service lands). 409 `fact_unbound` / `fact_readonly` returned otherwise.
- BASIS_OBJECT/BASIS_COLUMN/BASIS_REFERENCE (BLUEPRINT_META, unbuilt) proceeds as specced; basis-mirror fields are excluded from binding candidates in the Fact Harness to avoid double-counting.
- Migration is planned in six phases with rough effort: Phase 0 (1–2 days), Phase 1 (2–3 days), Phase 2 (4–5 days), Phase 3 (3–4 days), Phase 4 (2–3 days), Phase 5 ongoing (~1 day each consumer); Phase 6 is later/optional.
- Total estimated time to end of Phase 4: roughly 3 weeks.
- Rule established from Phase 2 forward: any new feature reading a project fact from labels fails code review.
