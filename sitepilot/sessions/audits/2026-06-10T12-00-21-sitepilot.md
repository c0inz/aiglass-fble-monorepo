## Decisions made

The investigation confirmed that project properties and property containers in SitePilot operate on a cascade/live/global model — no snapshot or per-project override mechanism exists. This is a factual determination from code analysis, not an operator decision.

The Properties tab render query in `routes/projects.js` (lines 829–890) performs a real-time JOIN against the live `am_field_spec` and `am_property_group` tables, meaning any change to a field definition or container assignment is immediately visible on every project's Properties tab.

The property CRUD API in `routes/settings.js` (lines 1930–2044) mutates shared global `am_field_spec` rows, not per-project rows. Creating, editing, or deleting a property affects all projects simultaneously.

Property deletion is blocked at the API layer if any `ai_field_entry` rows reference the spec — stricter than the FK cascade constraint would require.

`ai_field_entry` rows are created lazily (on first value save), not eagerly at project creation.

If a property's `data_kind` is changed (e.g., TEXT to NUMERIC), existing `ai_field_entry` rows are not migrated — they retain the old content column while the spec now expects a different column.

No admin UI exists for changing property-to-container (group) assignments at runtime; that mapping is set only during initial data load.

No migration or coercion occurs. This is a latent bug or risk with no corresponding milestone in the canvas.

The investigation noted that the settings UI (`views/settings/properties.ejs`) displays the "Group" column read-only with no edit capability, and group assignments can only be changed via direct database operations or the Coperniq load script. This represents a gap in admin tooling.

The `am_field_spec_group` and `am_property_group` tables use `ON DELETE CASCADE` on both foreign keys, meaning deletion of a group would silently remove all field memberships for fields in that group, immediately affecting all project Properties tab layouts.
