## Thoughts not yet in the canvas

- A sequencing recommendation was produced: extract `lib/shape/` over the existing id-addressed read/write first (pure refactor, no schema change), then run a blueprint dedup-merge pass on `am_field_spec`, then add key columns with deterministic collision-suffixing and per-blueprint-version unique indexes built `CONCURRENTLY`, and only then build the Agent API thin skin. The design doc's proposed build order was called out as wrong.
- A prerequisite dedup-merge pass on `am_field_spec` is required before any slug-uniqueness index can be built. Two duplicate `internal_name` values were found on the production database (`jurisdictionId` and `inverter_count_1`), each appearing across hundreds of ventures. This dedup work is not mentioned in the design doc.
- The design's "one write path" framing was identified as marketing rather than architecture: envelope mutations (status, assignee, dates, step transitions) must go through typed workflow endpoints, not `writeValues`. The doc conflates the two.
- `writeValues` needs a defined behavior for the case where a key resolves to more than one row (e.g., reject with 409 or merge). This is not specified in the design.
- The backfill of `def_internal_name` onto form-field instances should be batched off-peak, idempotent (`WHERE def_internal_name IS NULL`), and tracked in `lib/tenant-provisioner.js` as well as the migrations table, to cover both BRI0/SND0/MAS0/MAS1 live tenants and future provisioning.
- A "side-effects and transactional context" section is absent from the design. `writeValues` must run within the same tenant transaction and `requestStore` context as the current UI poster, and must return visibility and automation deltas — because visibility rules and automations are resolved live (not frozen), per the versioning doc.
- The design doc incorrectly lists `ai_location` (sites) and `ai_remark` (comments) as resources with `fields[]` arrays. Production schema shows `ai_field_entry` has no `location_ref` or `remark_ref` anchor column; sites carry no EAV and comments store a single `content` column.
