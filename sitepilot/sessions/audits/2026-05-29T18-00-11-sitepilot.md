## Thoughts not yet in the canvas

The review surfaces a Critical finding that the upload stored-XSS defense-in-depth header (Content-Disposition: attachment and X-Content-Type-Options: nosniff) was placed on the `/uploads/*` static route, which is not the actual serving path for uploaded attachments. The real serving routes are `/attachments/:id` (routes/attachments.js) and legacy `/files/*`. The `/attachments/:id` handler serves files inline with a MIME type sourced from `req.file.mimetype` — the client-supplied Content-Type at upload time — making it attacker-controllable. No `Content-Disposition: attachment` is set on this path. The global `nosniff` header at server.js:107 does not mitigate a case where the server explicitly declares `text/html` as the content type. The extension filter (the primary layer) does function correctly on upload, but a file uploaded with a benign extension and a spoofed `text/html` content type passes the filter and is later served inline as HTML.

A pre-existing privilege-escalation path exists in the worker-create handler (routes/settings.js): a caller with only `workspace.settings edit` permissions can supply arbitrary `role_ids` including the SUPERADMIN role ID, creating a new worker with SUPERADMIN privileges. This was present before the current branch but was noted as becoming more impactful because the fail-closed changes make roles the determinative access gate for the first time.

The `rolesAssigned` counter in settings.js increments optimistically before confirming a row was affected, though the FK constraint on `ai_worker_role.role_ref` makes this benign in the current schema.

The `requireMetaAccess` middleware passes requests with non-numeric IDs straight through to the route handler with no permission check applied (`isNaN(targetId) → next()`). This is pre-existing and unchanged, but any route handler relying solely on this middleware for authorization is bypassable by sending a non-numeric `:id`. A follow-up audit of meta routes was recommended.

A cosmetic inconsistency was noted: `requireSettingsAccess` uses the new shared `denyResponse` helper for the empty-roles denial path but retains an older inline 403 block for the permission-level denial; the other two gates were not similarly deduplicated.
