## Decisions made

The operator (owner/senior dev) requested a complete, opinionated architectural design for a canonical project-facts layer. The assistant produced a full design called "The Facts Ledger" with the following decided elements:

- Architecture name and thesis committed: promote a registry-governed canonical store ("Facts Ledger") as the system of record for project-level facts; all capture surfaces (forms, WOs, properties) become declared editors over it.
- Three new tables decided: `am_fact` (the registry with immutable `api_name`), `am_fact_binding` (bindings declaring which capture fields feed which facts), and `ai_fact_value` (typed row-per-fact storage with provenance columns).
- Two storage tiers decided within the registry: `storage='column'` for facts that already have real `ai_venture` columns (e.g., `contract_value`, `capacity_kw`), and `storage='fact_value'` for all others.
- The five `ai_venture` money-column ambiguity is resolved by registry designation: one fact is seeded per canonical money concept; others are not initially registered.
- `mode='mirror'` bindings on `am_fact_binding` subsume and replace the unbuilt `basis_object/basis_column/basis_reference` proposal from `docs/BLUEPRINT_META_FIELD_SPEC.md`. That spec is not to be built separately.
- Workflow-meta keys decided as companion migration 164: add `internal_name` to `am_stage_blueprint` and `am_assignment_blueprint`, seeded for known templates, killing phase-label and WO-title string-matching at the root.
- Migration numbering decided: `163_facts_ledger.sql` for the main schema, `164_workflow_blueprint_keys.sql` for workflow keys.
- Starter pack of ~25 seeded facts decided for solar-EPC tenants (including `installation_start_date`, `installation_completed_date`, `system_size_kw`, `pto_granted_date`, `ntp_date`, `site_survey_date`, `interconnection_approved_date`, `signed_contract` as FILE).
- Single writer discipline decided: only `lib/facts/` touches `ai_fact_value`.
- `v_project_facts` read view decided as a generated SQL view emitted by `scripts/facts/regen-view.js`, regenerated whenever the registry changes.
- Three declaration surfaces decided: a new Settings → Data → Project Facts page; the existing field-editor inspector (adding an "Updates project fact…" dropdown); and the workflow editor (event bindings for date facts).
- "Apply historically" backfill offered on binding creation, re-runnable and idempotent, implemented as `scripts/facts/apply-binding.js`.
- Historical survivorship rule for one-time backfill decided: venture column > newest form entry > stale Coperniq venture-scope EAV row.
- Agent API new endpoints decided: `GET /api/v1/meta/facts`, `GET /api/v1/projects/:id/facts`, `GET /api/v1/projects/:id?include=facts`, `PATCH /api/v1/projects/:id/facts`. The PATCH endpoint is designated as the Agent API's first write endpoint and will use the shipped-but-dead `am_api_idempotency` machinery (migration 159).
- Six-phase migration plan decided with rough effort estimates, each phase independently shippable.
- Weekly reconciliation job (diff bound entry values vs canonical) decided as the "honesty tax" enforcement mechanism.
- Blueprint version-copy code decided to copy live bindings forward on Lock Version bumps (migration 160 pattern), tested in Phase 2.
