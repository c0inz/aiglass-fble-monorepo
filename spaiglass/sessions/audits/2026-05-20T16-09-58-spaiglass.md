## Thoughts not yet in the canvas

Two bugs were diagnosed in the terminal session picker flow:

Bug 1: When a user selects a Terminal (claude-cli) session from the session picker, conversation history does not load. Root cause: `ChatPage.tsx` `getEncodedName()` searches the registered projects list (`/api/projects`) to find the encoded project name, but the terminal session's recorded `cwd` is often not in that list, so the lookup returns null and `useAutoHistoryLoader` skips the fetch. The picker already has the correct `encodedProject` value in `ClaudeSessionRow.encodedProject` (`shared/types.ts:70`) but `SessionPickerModal.tsx` does not pass it through the URL.

Bug 2: After selecting a terminal session, the user cannot reliably change directories and the page operates with an incorrect "current project" context. Root cause: `SessionPickerModal.tsx` rewrites the `cwd` query param to the terminal session's cwd but leaves the URL segment pointing at the old project. All URL-derived state (`__SG.project`, `__SG.segment`, AgentSwitcher, recent-agents recording, role resolution, page header, WebSocket tunnel routing) continues to reference the old project, while the working directory diverges. This produces a stale project reference that disrupts subsequent navigation.

Proposed fix #1: Add `encodedProject` as a URL param in the picker; have `ChatPage.getEncodedName()` check the URL param before falling back to the project list lookup.

Proposed fix #2 (additive to #1): When a terminal session's cwd is not a registered SpAIglass project, register it via `POST /api/projects/register` before navigating, then navigate to the real segment for that path. This aligns the URL with reality and unbreaks WS routing, the project switcher, and the breadcrumb. This approach adds an HTTP call before nav and creates a new `~/.claude.json` entry.
