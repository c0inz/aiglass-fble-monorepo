The operator reported that selecting a terminal session in the session picker does not land them in the correct directory, does not load terminal conversation history, and appears to interfere with subsequent directory changes. The assistant diagnosed two related bugs:

1. Terminal session history does not load because `ChatPage.tsx` `getEncodedName()` resolves the encoded project name by searching the `projects` array (populated from `/api/projects`), which only includes paths registered in `~/.claude.json` and filtered to non-internal paths. A terminal session's recorded `cwd` is often absent from that list, causing `useAutoHistoryLoader` to skip the fetch and leave scrollback empty. The fix is to pass `encodedProject` from `ClaudeSessionRow` through the URL in `SessionPickerModal.tsx` and check it in `ChatPage.getEncodedName()` before falling back to the project lookup.

2. The session picker rewrites the `cwd` query param to the terminal session's cwd while keeping the URL segment pointing to the previously active project. After navigation, the page operates with the terminal session's working directory while URL-derived globals (`__SG.project`, `__SG.segment`) reflect the old project. This causes stale state in the agent switcher, recent-agents recording, role resolution, page header, and WebSocket routing. The full fix requires registering the terminal session's cwd as a SpAIglass project (POST `/api/projects/register`) on pick, then navigating to the real segment for that path.
